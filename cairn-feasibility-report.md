# Cairn — Feasibility Report

> Skeptical principal-engineer review of `PRD-cairn.md` (v1.0, June 2026).
> **Method:** 20 technical assumptions decomposed from the PRD, each researched against current (2025–2026) primary sources by a parallel agent; every High-risk/Infeasible verdict adversarially re-checked by an independent verifier; alternatives researched for each risky/blocked item (30 agents, ~1.65 M tokens, 468 tool calls). Three highest-risk assumptions were **spiked with throwaway code in `spikes/`** (run on this machine). Confidence and license are recorded per row.

## Verdict

**GO-WITH-CHANGES.** Cairn is technically buildable: of the 20 assumptions, 2 are Feasible and 16 Feasible-with-caveats — only **2 are High-risk** (and *zero* survive as Infeasible after verification). Critically, the **permissive-license audit passes in full**: every required capability has a strong MIT/Apache/BSD option and *nothing* is forced onto AGPL/GPL (the two AGPL traps — MuPDF for PDF annotation, espeak-ng for TTS — both have clean permissive escapes). The two genuine problems are not missing technology but planning: the **16-week solo timeline is 3–6× too optimistic** against every comparable project (realistically 12–24 months as scoped), and the **"local models for everyone" promise breaks on the 8 GB hardware the PRD explicitly names** for its zero-setup persona. Adopt the descope + hardware-gating changes in §Revised recommendation and Cairn is a sound build; ship it exactly as written and it stalls on timeline and disappoints non-technical students on weak machines.

## Spike findings

All spikes run on this machine: **macOS, 18-core / 128 GB RAM, Node 24.14, sqlite 3.51.** This is *far* above the PRD's "8–16 GB student hardware" target — model **latency numbers below are best-case ceilings and not representative of the persona.** Spike code lives in `spikes/`.

**Why these three (and not the two final High-risk items):** spikes target the highest-risk *code-testable* uncertainties. The two remaining High-risk assumptions are deliberately *not* spiked here — A8 (8 GB-hardware latency) is not measurable on this 128 GB box, and A19 (solo-timeline realism) is a research question, not code. Note also that each spike below proves a *specific* sub-claim, not the whole assumption: §2 proves the coordinate math, not the annotation UI; §3 proves single-turn tool-calling, not the multi-step Agent loop. The residual untested parts are called out per spike and are why those features are deferred in §Revised recommendation.

### Spike 1 — sqlite-vec: brute-force scan, not ANN (the key correction) — `spikes/sqlite-vec/perf.js`

Installed `sqlite-vec@0.1.9` + `better-sqlite3@12.10`. Built a `vec0` table of 768-dim float32 vectors (nomic-embed-text dimension), timed `MATCH ... ORDER BY distance LIMIT 8` at increasing row counts (file-backed DB, WAL):

| rows | insert (s) | query p50 | query p95 | db file |
|---|---|---|---|---|
| 10,000 | 0.2 | 3.5 ms | 3.9 ms | 30 MB |
| 50,000 | 0.8 | **18.5 ms** | 19.6 ms | 148 MB |
| 100,000 | 1.0 | 37.3 ms | 38.6 ms | 297 MB |
| 200,000 | 2.1 | 74.5 ms | 77.9 ms | 593 MB |
| 500,000 | 5.8 | 181.7 ms | 185.8 ms | 1,480 MB |

**Finding:** latency grows **strictly linearly** with row count (~52× time for 50× rows) and `EXPLAIN QUERY PLAN` shows `SCAN vec_items VIRTUAL TABLE` — i.e. sqlite-vec does a **brute-force linear KNN scan; there is no HNSW/IVF ANN index.** The PRD's repeated "ANN search" language is therefore inaccurate.

**Implication:** at the PRD's stated v1 scale (~50k chunks ≈ 1k notes + 50 PDFs) this is a **non-issue — ~18 ms, imperceptible.** But the cost is O(n): the P3 "years of accumulation" persona reaching 200k–500k chunks sees 75–180 ms *on this fast box*, and noticeably worse single-threaded on student hardware. On-disk cost ≈ 3 KB/vector. **Verdict: Feasible-with-caveats** — fine for v1, but the docs/claims should say "brute-force KNN," and a scale ceiling (~low-hundreds-of-thousands of chunks) exists. Mitigations: metadata pre-filtering to shrink the scan, or migrate to a true ANN (LanceDB/hnswlib) only if vaults grow large.

**Separately (the real Electron risk, NOT tested here, doc-verified — see assumptions table):** code-signing/notarizing a loadable SQLite extension `.dylib` inside a hardened-runtime macOS Electron build, and `electron-rebuild` ABI churn per Electron version. The perf number says nothing about this; it is a distinct row.

### Spike 2 — pdf.js coordinate mapping is sound, incl. area-annotation re-anchoring — `spikes/pdfjs/coords.mjs`

`pdfjs-dist@6.0.227`. Built a known-coordinate fixture with `pdf-lib` (text "HIGHLIGHT ME" at PDF (100,700); a diagram rectangle at (200,300,380,420)), then:
- **Text-highlight (easy) path:** `getTextContent()` returns each item's `transform` matrix + width/height → text run recovered at PDF (100,700)→(273,724), **exact match** to where it was drawn.
- **Area-annotation (the real risk) path:** simulated a user box drawn at **1.5× zoom** → `convertToPdfPoint` to persist PDF-space coords → **recovered (200,300,380,420) exactly** → `convertToViewportRectangle` re-projected at 1×/2×/3×, with width/height scaling linearly (180/120 → 360/240 → 540/360 px), and survived a 90° page rotation (non-degenerate rect).

**Finding:** the viewport↔PDF transform is **clean and reversible at any zoom/rotation**; storing rects in PDF coordinate space is the correct, zoom-stable persistence model, and the sha256 sidecar key is trivial. The hard part the PRD risk-table feared (coordinate stability across zoom for area annotations) is **largely retired.** Residual risk is **UI plumbing, not math**: DOM overlay positioning over the text layer, multi-span selection highlighting, hit-testing, retina `devicePixelRatio`. PDFs are fixed-layout so text *reflow* is a non-issue. **Verdict: Feasible** (effort is in the interaction layer). Not GUI-testable headlessly — the interactive drawing/rendering still needs a real-window test.

### Spike 3 — Ollama: embed + small-model tool-calling both work (capability, not latency) — `spikes/ollama/capability.mjs`

Installed `ollama@0.30.8`, pulled `nomic-embed-text` (274 MB) + `llama3.2:3b` (2.0 GB).
- **Embed round-trip:** 768-dim vectors returned; a 3-doc retrieval sanity check ranked correctly — query *"how do plants make energy from sunlight?"* → photosynthesis **0.77** > mitochondria 0.61 > quicksort 0.33. Embedder discriminates meaning as needed for RAG.
- **Tool-calling (OpenAI-compat `/v1`):** over 5 trials, llama3.2:3b emitted a valid `search_notes` tool call **5/5** with valid JSON args **5/5**. A 2-step loop (call → feed tool result → final answer) produced a grounded answer using the injected content. **Better than expected for a 3B model.**

**Caveats that keep this Feasible-*with-caveats*, not Feasible:** (1) this is the *easy* case — one tool, strong system prompt, unambiguous query; it is **not** evidence of reliability across 6 modes, multiple tools, and 25-step Agent loops where small models degrade. (2) The model coerced `k` to the **string** `"1"`, violating the integer schema — local models need server-side arg-validation/coercion. (3) Latency here is meaningless for the persona (this box ≠ 8–16 GB). (4) License nuance: the *strong* small tool-callers are Llama-family (**Meta Community License, not OSI-permissive**); the permissive alternative is **qwen3 (Apache-2.0)** — relevant only if Cairn ever *bundles* a model (it doesn't; users pull their own).

## Blockers & alternatives

There are **no NO-GO technical blockers.** After adversarial verification, two assumptions are High-risk and two carry license flags with clean escapes. Each below states the recommended fix and why.

### B1 — Solo 16-week timeline (Assumption 19, High-risk) — *the headline issue*
**Problem:** The plan asks a part-time student to ship the *union* of scopes that took multi-person teams 1.5–3 years each. Khoj = 2 co-founders + an intern over ~3 years; Open Notebook = ~1.5 years and is *only* a web app (no Electron packaging, no annotation, no agent loop); Reor — the closest Electron + notes + local-RAG analog — ran ~18 months, then **stalled and was archived**, and still only covers the editor+RAG subset. A realistic multiple is **3–6× → 12–24 months** as scoped, longer around coursework. (The first research pass called this *Infeasible*; the verifier downgraded to *High-risk* because it is reachable *with aggressive descoping* — see §Revised recommendation.)
**Fix:** Cut v1 to a vertical slice and stage the rest (concrete cut-list below). Do **not** try to rescue the timeline by lifting code: the nearest-scope bases (Reor, AppFlowy) are **AGPL-3.0** — legally off-limits under the permissive policy *and* archived/wrong-stack. The one real accelerant is **building as an Obsidian plugin** (inherits editor/graph/wikilink/vault + cross-OS packaging for free) — but that contradicts PRD §4's standalone-app identity, so it's a strategic choice, not a drop-in.

### B2 — Local models on 8 GB hardware (Assumption 8, High-risk)
**Problem:** The assumption is a conjunction — 8B chat **+** embeddings **+** 8–16 GB **+** usable latency — and it fails on the *lower half* of the range the PRD itself names. An 8B Q4_K_M model needs ~7.5 GB for weights alone; with the OS, a concurrently-loaded embedder, and KV cache it does **not** fit in 8 GB unified RAM, and on CPU-only laptops an 8B runs at ~3 tok/s — so the "<10-min study-guide+quiz" metric becomes 6–22 min of decode alone. The danger is the **first-run flow auto-offering the 8B pull to *everyone*, including the P2 "zero-setup non-technical student"** on exactly the hardware where it thrashes. (Verifier confirmed High-risk on corrected grounds — not refuted.)
**Fix (selection-policy change, not a library swap — keep Ollama, MIT):** probe RAM/GPU at first run and **gate the 8B pull**; default 8 GB / no-GPU machines to a **3–4B chat model** (Qwen3-4B, Apache-2.0, ~15–20 tok/s on an M1 Air) + a small embedder, and set quality expectations honestly. Optional substrate upgrades that preserve no-server: `node-llama-cpp` (in-process, auto-adapts to hardware) or `llamafile` (single-file, zero-install). Lean on RAG grounding to compensate for the smaller model.

### B3 — PDF annotation editing if pdf.js proves too weak (Assumption 4, ⚠ license flag)
**Problem:** The seed assumption named "PDFium via napi" and MuPDF — but the native NAPI PDFium binding (`node-pdfium`) is **dead** (v0.0.1, ~11 yrs, build-fails on modern Node), and **MuPDF/mupdf.js is AGPL-3.0** (hard block).
**Fix:** If pdf.js's overlay approach is insufficient, adopt **EmbedPDF** (`@embedpdf/*`, **MIT**, PDFium-WASM) — the only permissive option that ships a real annotation *editing* layer and persists into the actual PDF via `saveAsCopy`. Trade-off: young single-vendor OSS (~4.2k stars) and WASM bundle/memory overhead. Commercial SDKs (Nutrient/PSPDFKit, Apryse) are best-in-class but proprietary and ~$10k–220k/yr — excluded on policy/cost. **My spike (§2) shows pdf.js's own coordinate model is sound, so EmbedPDF is a fallback, not a day-1 need.**

### B4 — Tier-2 local TTS hidden GPL dependency (Assumption 17, ⚠ license flag)
**Problem:** Both practical JS paths phonemize through **espeak-ng (GPL-3.0)**: Piper's MIT repo was archived Oct 2025 and moved to a GPL fork; Kokoro's `kokoro-js` pulls `phonemizer`, which bundles espeak-ng as WASM. The Apache/MIT labels cover only the wrappers, not the bundled GPL code that would ship inside the app.
**Fix:** Use **piper-plus** (`ayutaz/piper-plus`, **MIT**, espeak-free via `g2p-en`/CMUdict) — the only option satisfying permissive + espeak-free + runs-in-Electron-today. This is deferred Tier-2 behind a flag, so **non-blocking now**, but must be resolved before that feature ships. (Ship the dialogue *script* generation first — it has study value without audio.)

### Lower-severity caveats worth budgeting (not blockers)
- **Tauri portability claim is misleading (A1):** the React renderer ports, but the Node main-process services (better-sqlite3, isomorphic-git, chokidar) have no Tauri equivalent and need a Rust rewrite. Keeping a "thin IPC layer" does not make a port cheap; the backend rewrite is the cost. Treat Tauri as a *possible v2 rebuild*, not a guaranteed escape hatch.
- **Citation-after-edit (A12)** is the W3C fuzzy-anchoring problem — solvable (Hypothesis ships it) but no correctness guarantee and the off-the-shelf libs are archived/stale, so vendor a fuzzy layer on `diff-match-patch`.
- **Graph view (A14)** needs a WebGL renderer + Web-Worker/precomputed layout; naive main-thread d3-force janks at 1000+ nodes.
- **isomorphic-git (A15):** no `git clean`, so the app must delete Agent-created files itself for a truly byte-identical revert; never mix iso-git with system git (autocrlf divergence).
- **safeStorage (A18)** stores only the *key* in the keychain (app persists ciphertext itself) and **silently falls back to near-plaintext on Linux** with no secret-service backend — detect and warn.
- **Mermaid mindmap (A16)** is still "experimental"; Anthropic structured outputs reject recursive schemas, so use a fixed-depth tree schema + client-side validation.

## Revised recommendation

**Build it — as an explicitly descoped v1, on Electron, with two policy changes.** The architecture and library choices are sound and fully permissive; the PRD's own instincts (sidecar JSON, plain files, per-step approval, tiered Studio) are right. Make these changes:

**1. Re-scope v1 to a vertical slice (fixes B1).** Ship the dogfood loop end-to-end, defer the breadth:
- **Keep (v1):** vault shell + CodeMirror editor + wikilinks/backlinks; RAG core + citations (the heart); Ask + Tutor; **text** highlights + sidecar + `[[pdf:]]` links + selection→Tutor; Study Guide + Quiz; git snapshots; **macOS only**.
- **Defer (v1.5+):** **area/rectangle annotations** (PRD already says cut these first), **Agent mode** (the autonomous multi-step loop is the riskiest × least-proven-on-local part — ship the gated write-modes first), Synthesize/Plan, the other 4 Studio generators, **interactive** mind-map (ship Mermaid export only), hybrid retrieval, Windows/Linux packaging, and **all of Tier-2 audio**.
- This turns "16 weeks for the union" into "16 weeks for a usable core," which matches what comparable solo efforts actually achieved.

**2. Make local-model selection hardware-aware (fixes B2).** Replace "offer qwen3:8b to everyone" with a RAM/GPU probe that gates the 8B pull and defaults weak machines to Qwen3-4B + a small embedder; show honest quality/latency expectations. Prefer **Qwen3 (Apache-2.0)** over Llama as the default weights to keep the model story OSI-clean.

**3. Correct four PRD statements before they mislead implementation:**
- sqlite-vec is **brute-force KNN, not "ANN"** (fine at v1 scale ≈18 ms/50k; plan a true-ANN migration path — hnswlib/LanceDB — only if vaults exceed low-hundreds-of-thousands of chunks).
- Use **Electron `safeStorage`**, not keytar (archived).
- Treat the **Tauri port as a possible v2 rewrite**, not a free port behind IPC.
- For local Agent-mode reliability, prefer **grammar-constrained decoding** (llama.cpp GBNF / `node-llama-cpp`) and lean on the PRD's **mandatory per-step approval as the safety net** — it converts flaky multi-step tool loops into "more rejected steps," not a corrupted vault, which is exactly the right mitigation.

**Net:** No NO-GO. With the descope and hardware-gating, every remaining assumption is Feasible or Feasible-with-caveats and the license policy holds end-to-end. The spikes earn their keep precisely by testing the load-bearing sub-claims: they **retire the PDF coordinate-mapping risk**, **correct the PRD's "ANN" claim** (sqlite-vec is brute-force, fine at v1 scale), and **confirm single-turn local tool-calling**. What they deliberately did *not* test — the multi-step 25-step Agent loop and the annotation editing UI — is exactly the residual that keeps both features deferred rather than shipped in v1.
